# PROJECT MEANING MAP — Digital Code / Zerkalo

_Last updated: 2026-04-29_

This document explains what the project is, what it is not, and how the current product pieces should work together.

## 1. Core idea

Digital Code / Zerkalo is a premium self-knowledge product built around date-based calculation, structured interpretation, and a high-touch reflective document.

The product should feel like:

- an accurate internal map;
- a mirror of personality patterns;
- a practical route for action;
- a premium personal file;
- a living bridge between calculation and human language.

It should not feel like:

- a generic numerology site;
- an entertainment quiz;
- a therapy product;
- a fortune-telling product;
- a cheap AI content generator.

## 2. Current product ecosystem

```text
Zerkalo web app
  -> First Mirror / Personal Myth
  -> lead capture
  -> owner notification
  -> manual conversation
  -> Big Research document
  -> optional Telegram follow-up
```

## 3. Zerkalo's role

Zerkalo is the front door.

Its job is to create the first emotional and intellectual trust:

- the user enters a date or answers a few reflective questions;
- the system gives a short but precise result;
- the user feels that the product is not generic;
- the user requests the deeper product.

Zerkalo should not try to deliver the whole paid value for free.

## 4. Telegram bot role

The bot is the return channel.

It should:

- explain the product briefly;
- open Zerkalo through a button;
- later deliver results;
- support simple follow-up;
- notify the owner or operator about leads.

It should not become the core interface before the web flow is validated.

## 5. Notion role

Notion is the project headquarters.

It should store:

- strategic decisions;
- product architecture;
- canon and methodology notes;
- launch plans;
- quality rules;
- handoff pages for agents;
- premium document standards.

Notion should not be treated as a dumping ground. It should be organized as a decision and methodology layer.

## 6. GitHub role

GitHub is the execution source of truth.

It should store:

- production code;
- deploy files;
- docs;
- agent handoff files;
- prompts and QA gates;
- changelog decisions that affect the app.

Any agent working on code should begin with `docs/MASTER_CONTEXT_HANDOFF.md`.

## 7. Paid product: Big Research

Big Research is currently the main monetizable product.

It should remain manual or semi-manual until real demand is proven.

Core promise:

```text
A precise personal research file based on calculation, matrix, compound numbers, current period, and living interpretation.
```

Primary value:

- not volume;
- not mystical certainty;
- not automation;
- but exact personal recognition and practical direction.

## 8. Personal Myth layer

Personal Myth is an additional emotional entry point.

It should work as a literary reflective mirror:

- image;
- inner tension;
- hidden resource;
- new view;
- one step.

It must not be positioned as therapy or diagnosis.

## 9. Strategic priority

The next project phase is not feature growth. It is demand validation.

Main questions:

1. Does the user leave a request after First Mirror?
2. Does the user leave a request after Personal Myth?
3. Which entry point converts better?
4. What price feels realistic?
5. Can the first Big Research files create strong feedback?

## 10. Current freeze

Keep focus. Do not expand into:

- compatibility;
- automatic PDF;
- account system;
- full payment integration;
- mass ad traffic;
- complicated CRM;
- n8n as core.

Only add automation after real leads appear.

## 11. Success definition

The project succeeds at this stage if it produces:

- 5–10 real testers;
- first qualified leads;
- 1–3 manually prepared Big Research files;
- feedback proving that people want the deeper product;
- a clearer price and offer.
